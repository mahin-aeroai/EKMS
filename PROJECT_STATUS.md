# MMDI ONE — Project Status

Last updated: 15 July 2026

This file exists so a new chat session (or a new contributor) can pick up this
project without re-deriving context. Read this before making changes.

## What MMDI ONE is

An AI-native enterprise operating platform for MMDI (a packaging/printing
manufacturer). Built by Srinivas and his son Mahin as a knowledge-share
project. Currently: a Next.js app implementing a full component design
system plus 26 "Intelligent Workspace" modules covering the whole MDI-ONE
navigation tree, with 20 of those modules wired to a live Supabase backend.

- Live demo: https://ekms.vercel.app
- Repo: https://github.com/mahin-aeroai/EKMS (main branch, auto-deploys to Vercel)
- Stack: Next.js 16 (App Router) + React 19 + TypeScript + Tailwind CSS v4 + Supabase

## Current state (done, verified working)

1. **Design system**: 42 reusable UI components (`src/components/ui/`), 3 themes
   (Light/Dark/Enterprise), full style-guide showcase app under `/components/*`
   and `/foundations`.
2. **4 flagship workspaces** — Customer, Machine, Raw Material, Project — each
   using the full 6-tab Universal Workspace Pattern (Overview / Insights /
   Timeline / Documents / Relationships / Activity). These use a
   Server Component + Client Component split:
   - `src/app/workspaces/{customer,machine,raw-material,project}/page.tsx` —
     async Server Component, `export const dynamic = "force-dynamic"`, fetches
     initial data from Supabase server-side.
   - `src/components/workspaces/*.tsx` — `"use client"` component handling
     tabs, optimistic UI, and real Supabase writes (comments insert, approval
     status update).
   - Only Overview + Activity tabs are wired to real data; Insights/Timeline/
     Documents/Relationships remain sample content (deliberate scope decision).
3. **22 lighter workspace modules** covering the rest of the MDI-ONE IA
   (Executive, Customers, Operations, Manufacturing, Knowledge, People,
   Finance, Compliance, Administration groups in the sidebar nav). Pattern:
   header + stat row + one AI insight card + one real domain widget (table,
   kanban, calendar, tree, etc.) — not the full 6-tab pattern.
   - **16 of these are wired to real Supabase data** (simple `useEffect` +
     `useState` fetch, no Server/Client split needed): CRM, Quotations,
     Contracts, Production, Maintenance, Installation, Inventory, Procurement,
     Suppliers, Documents, Drawings, SOPs, Lessons Learned, People,
     Compliance, Administration.
   - **6 remain sample/hardcoded data** (deliberately, since they're
     cross-module aggregation/dashboard pages with no single natural backing
     table): Command Center, AI Copilot, Analytics, Costing, AI Knowledge,
     Finance.
4. **Backend**: Supabase project `mahin-aeroai's Project`
   (`https://vzyrvzgtjcodxkjydxxn.supabase.co`), free tier. Single shared
   browser-safe client at `src/lib/supabase.ts` using the anon/publishable
   key (safe to expose — protected by RLS, not a secret). All row-type
   TypeScript interfaces live in this same file.
5. **RLS policies are wide open** (`using (true)` / `with check (true)`) on
   every table — this is temporary and flagged in the SQL files. **There is
   no authentication system yet.** Anyone with the anon key can read/write
   every table. Fine for an internal demo, not fine for production.
6. Build and lint both verified clean on every change (`npm run build`,
   `npm run lint` — 0 errors). All routes render as expected (static for the
   22 lighter modules, dynamic/force-rendered for the 4 flagship workspaces).

## What's NOT done yet (known gaps)

- **No authentication.** This is the biggest open item — RLS policies need
  real user-scoped rules once auth exists.
- The 6 aggregation dashboards (Command Center, AI Copilot, Analytics,
  Costing, AI Knowledge, Finance) are still hardcoded sample data.
- The 4 flagship workspaces only have Overview + Activity wired to Supabase;
  the other 4 tabs per workspace are still sample content.
- No real AI/LLM integration anywhere yet — all "AI insight" cards are
  static copy, not generated by a model.
- No file upload / document storage wired (Documents, Drawings, SOPs pages
  read metadata rows from Supabase but don't handle actual file storage).
- No tests.

## Key files to know

- `src/lib/supabase.ts` — Supabase client + every row-type interface. Read
  this first before touching any workspace page.
- `src/components/AppShell.tsx` — top nav, sidebar (all IA groups), command
  palette, AI assistant drawer wrapper.
- `src/app/workspaces/*/page.tsx` — one folder per module. Flagship ones
  (`customer`, `machine`, `raw-material`, `project`) delegate rendering to
  `src/components/workspaces/*.tsx`; the 22 lighter ones are self-contained.
- SQL schema files (not committed to the repo, delivered separately to the
  user and already run against the live Supabase project):
  - Customer workspace schema (customer_contacts, customer_comments,
    customer_approvals, etc.) — run first, earliest phase.
  - Machine / Raw Material / Project workspace schemas — run in the
    "wire 3 remaining flagship workspaces" phase.
  - `supabase-remaining-modules-schema.sql` — the 16-table schema for the
    lighter modules listed above. **Already run against production** as of
    15 July 2026.

## Working conventions established in this project

- Build/lint must happen in `/tmp/build/mmdi-one-design-system` (synced via
  rsync, excluding `node_modules`/`.next`/`.git`) — building directly in the
  mounted outputs folder throws `EPERM` on `.next` cache writes.
- I (the assistant) cannot handle GitHub credentials or push on the user's
  behalf — every handoff is packaged as a zip of only the new/changed files,
  with exact `unzip -o … -d .` + `git add -A && git commit && git push`
  instructions given to the user to run themselves.
- zsh does not enable `interactive_comments` — never include inline `#`
  comments in copy-pasteable multi-line shell command blocks given to the
  user.
- `react/no-unescaped-entities`: escape apostrophes (`&apos;`) in JSX text
  children, but NOT inside JSX string attributes (e.g. `title="What's..."`).
- Avoid `setState` directly inside `useEffect` bodies for lint compliance —
  either accept the `react-hooks/exhaustive-deps` disable comment (used
  throughout the 16 lighter modules' fetch effects) or use a lazy `useState`
  initializer.
- Kanban-style pages (Production, Procurement) reconstruct `KanbanColumn[]`
  client-side from flat DB rows via a `toKanbanColumns()` helper grouping by
  a `column_id` text field, using a fixed `COLUMN_ORDER` + `COLUMN_TITLES`
  lookup.
